WEB SECURITY

  • Teoria

    Lo scenario web è uno scenario con diversi strati: il browser si interfaccia, attraverso il protocollo HTTP, con il server web che fornisce l’accesso ad applicazioni dinamiche che eseguono per generare la risorsa richiesta, elaborando sia le risorse fornite dal cliente che quelle conservate lato back-end (all’interno dei database). IN questo contesto le vulnerabilità possono essere diverse:

    • Lato cliente = nell’interazione con il server e con l’esecuzione del codice sul client.
    • Protocollo = la sicurezza del canale di comunicazione, il trattamento dei dati errati di HTTP sugli endpoint, ovvero i dati che possono essere trattati in modo tale che client e server li interpretino in modo sbagliato.
    • Server = errore di gestione delle richieste HTTP da parte del server, errori sull’accesso o di interpretazione dei dati.

    Le principali tipologie di attacchi via web sono:

    1. Broken access control

    Raggruppa una varietà di problematiche di accesso non correttamente mediato dalle risorse (voglio che solo degli utenti accedano ad una determinata risorsa e per farlo è necessaria una catena di autenticazione, autorizzazione ecc). Questo problematiche si possono verificare attraverso l’esposizione di identificativi di oggetti (IDOR), lasciando l’accesso a tutti quelli che conoscono l’identificativo.

    Un caso particolare IDOR è il file disclosure, dove viene lasciato il riferimento ad un file. Il caso classico è quello che viene chiamato path traversal che si verifica quando attraverso un path (che può essere inserito dall’utente), modificandolo adeguatamente, si accede ad un altro ramo non indicato come per esempio: /home/maybe/deep/username/../../../../etc/passwd (l’utente aveva inserito la prima parte del path, ma dalla radice scendo all’interno di un altro file).

    1. Cryptographic failures

    Le applicazioni web manipolano grosse quantità di dati sensibili che devono essere protetti in ogni fase del ciclo di vita dell’applicazione web. I principali problemi che si possono verificare sono:

    • Alcuni dati non vengono riconosciuti come sensibili.
    • Non si individuano tutte le copie del dato da proteggere (DNR con lo scaricamento lato browser di file multimediali/musicali, backup non protetti).
    • Non si proteggono tutte le fasi di vita di un dato sensibile.
    • Non si utilizzano metodi di protezione adeguatamente robusti.

    Un esempio di una copia del dato da proteggere che potrebbe causare problemi.

    Il database è correttamente protetto e il canale di comunicazione sicuro. Il sistemista ha deciso di loggare tutti gli errori. Può succedere che quando si verifica un errore di comunicazione tra il server e la banca, venga loggato un errore contenente il numero di carta dell’utente.

    1. Injection

    L’iniezione si verifica con l’invio di input non fidati ad un interprete che provocano l’esecuzione di comandi che possono violare alcuni requisiti di sicurezza.

    -SQL injection

    L’esempio più importante è la SQL injection. SQL è un linguaggio interpretato e di conseguenza contiene una serie di caratteri speciali. Il programmatore non sa a priori come verrà interpretato il carattere speciale, in quanto si può essere sicuri di questo solo quando il comando viene passato all’interprete. Per evitare questo, SQL permette di pre-compilare le query.

    Un applicazione web chiede l’inserimento di username e password. Lato server viene costruito la query SQL con parti fisse e variabili (username e password). Inserendo alcuni username e password si possono avere alcuni effetti indesiderati.

    Nel primo caso la condizione WHERE è sempre vera e restituisce tutte le entry del database. Nel secondo caso si hanno due query: la prima fallisce, la seconda distrugge completamente il database.

    -Cross-site scripting

    Un’altra sotto-categoria dell’injection è il cross-site scripting, tipologia di iniezione di codice lato browser che può avvenire in diversi modi:

    • Resa possibili da applicazioni vulnerabili su siti in cui ti fidi.
      • Reflected XSS = Un esempio è un una pagina reflected. Un malintenzionato identifica un XSS all’interno di un’applicazione web e crea un codice javascript che originariamente non era presente. Attraverso tecniche di phishing, diffonde la pagina modificata appena creata confondendo l’utente e facendogli inserire le sue credenziali, nonostante sia già autentificato.
      • Stored XSS = Nella stored jnjection, l’attaccante inietta script sull’applicazione web conservate nel server, il quale le salva all’interno di un database. Quando la vittima fa una query al server, che comporta l’interrogazione del database, i dati vengono caricati dal database, inseriti come risultato di composizione della pagina e inviata all’utente.
    • Resa possibile da un’applicazione vulnerabile in esecuzione sul browser.
      • DOM XSS = La DOM injection è molto simile alla reflected, ma il codice che processa i dati non è sul server, ma all’interno della pagina stessa.
    1. Insecure design

    Finora tutte le insicurezze viste sono a livello implementativo. L’insecure design pone il punto di attenzione sulla security by design, dove la sicurezza deve essere presa in considerazione in ogni fase della modellazione del progetto. Nell’industria che fa uso di software si sta affermando il paradigma shift-left, dove i problemi di sicurezza vengono spostati verso sinistra nella catena di fasi di sviluppo software SDLC:

    L’esempio più comune è che molte applicazioni vengono installate con delle credenziali di default che poi non vengono modificate e rappresentano un punto di accesso per gli attaccanti.

    -Protocolli e header

    La security misconfiguration può verificarsi anche a livello di protocollo crittografici. Lato client è possibile inserire una serie di righe di header HTTP per consigliare o far sì che il browser si comporti in un certo modo. Alcuni degli header sono:

    • X-Frame-Options = per controllare l’inserimento di frame, che permettono di includere delle pagine all’interno di altre pagine.
    • X-XSS-Protection = tenta di filtrare reflected Cross Site Scripting.
    • Content-Security-Policy = può bloccare caricamento di risorse esterne rispetto alla pagina stessa.
    • X-Content-Type-Options = un attaccante può modificare il content-type, imponendo al browser web un determinato rendering, ma iniettando un'altra tipologia di dati (come un file binario).
    • Cache-Control
    • Strict-Transport-Security = impone ai browser di interagire col server solo con HTTPS.
    • Public-Key-Pins

    Una soluzione drastica contro questo approccio è il SOP (Same Origin Policy), che consente a pagine e script l’accesso a risorse provenienti dalla stessa origine. Il suo raffinamento è CORS dove si possono esprimere le diverse eccezioni per l’acceso a particolari risorse.

    -XML external entities

    XML permette di fare riferimento a entità esterne. Se è presente un parser XML vulnerabile, è possibile iniettare codice XML per lettura di file e attacchi di vario genere. Per mitigare questi effetti è necessario usare parser aggiornati e disattivare il supporto alle external entitity.

    1. Componenti software con vulnerabilità note

    Per esempi heartbleed.

    1. Problemi di identificazione e autenticazione

    In forte calo grazie all’utilizzo di framework standardizzati. In generale le applicazioni web fanno utilizzo di sessioni. Quest’ultima viene identificata attraverso lo scambio trasparente tra l’utente e il server di un token che identifica la sessione, ricevuto al primo accesso sul sito (il funzionamento di questo token è uguale a quello dei cookie, elementi che consentono di avere dello stato all’interno di un protocollo stateless). I problemi che si possono verificare per quanto riguarda l’identificazione possono essere diversi:

    • Anticipazione dell’attribuzione del token = l’attaccante in maniera proattiva avvia una sessione in modo tale da farsi consegnare un token dal server. Tramite tecniche di pishing convince l’utente a non autentificarsi, ma a lavorare sulla sessione da lui aperta. All’interno di questa di questa sessione fa inserire credenziali all’utente per poi rubarle e accedere al server con le credenziali rubate.
    • Cross-site request forgery = una vittima si autentica in un sito e viene convinta dall’attaccante a visitare anche il suo sito. Il sito dell’attaccante, di nascosto, manda un form alla pagina del sito acceduto dall’utente.
    • L’id è prevedibile, intercettabile (viaggia in chiaro e non attraverso HTTPS), non viene fatto scadere o non viene invalidato al termine della sessione.

    Per risolverlo è necessario non autentificare il cliente, ma ogni singola azione.

    1. Problemi di integrità del software e dei dati

    Questa problema si verifica includendo codice, dati e pezzi di infrastruttura che non sono al livello dei miei standard di sicurezza. Per quanto riguarda i dati, il maggiore problema è dovuto alla serializzazione in quanto si possono inserire, violando l’integrità del flusso, oggetti che non facevano parte del flusso iniziale.

    1. Log and monitoring failures

    Se vengono inserite troppe informazioni all’interno dei file di log, queste possono essere sfruttate da un attaccante.

    1. Server side request forgery

    Un utente inserisce una richiesta che comporta l’esecuzione da parte del server di una query con conseguente interrogazione di un database remoto che non era raggiungibile da parte del cliente dal browser.

    La prima vulnerabilità che si vuole testare è quella del brute force con un form di inserimento di login. Spesso questi attacchi si verificano attraverso la predisposizione da parte dell’attaccante di wordlist note o fatte ad hoc (vedi laboratorio precedente).     • Send to intruder = per vedere l’effetto di una serie di attacchi con l'inserimento di un particolare payload nella richiesta

    -(Remote) File inclusion

    Il file inclusion è un altro tipo di vulnerabilità che permette, a partire da una richiesta HTTP, una risorsa locale che inizialmente non era resa visibile, semplicemente modificando l'URL della richiesta HTTP (anche attraverso tecniche di path traversal). Lo stesso concetto si può applicare anche su risorse remote ma visibili da un entità intermedia.

    -Command injection

    Per testare la command injection è presente un form dove possiamo inserire un indirizzo IP su cui fare un ping (vedendo la configurazione il ping viene limitato a 4 secondi, ma nei sistemi unix andrebbe all'infinto). Intercettando la richiesta tramite burp e mandandola al repeater, posso inserire un cat /etc/passwd dopo l’indirizzo IP su cui viene fatto il ping. Visualizzando la risposta, si nota che la risposta contiene anche il contenuto del file specificato (il comando inserito però potrebbe causare problemi in quanto non è codificato in URL. All'interno di burp è presente un decoder/encoder per le conversioni. Se si converte il comando in URL e si inserisce nella richiesta HTTP precedente, si ottiene il medesimo risultato). Se si aumenta il livello di sicurezza, la configurazione della richiesta viene modificata in quanto vengono eseguite le seguenti sostituzioni:

    $substitutions = array(

    '&&' => '',

    ';'  => '',

    );

    Per questo motivo se proviamo ad eseguire lo stesso comando di prima non otteniamo lo stesso risultato in quanto ; è stato convertito in spazio. Per evitare questo problema si può rimuovere l'indirizzo IP e inserire il comando di pipe || cat /etc/passwd.

    -SQL injection

    In questa sezione si trova un form per l’inserimento di ID che porta alla formazione di una query SQL. La query viene completata in questo modo:

    $query  = "SELECT first_name, last_name FROM users WHERE user_id = '$id';";

    inserendo come id= ' OR '1'='1, la clausola WHERE è sempre verificata e di conseguenza ci vengono restituiti tutti gli utenti appartenenti al database.

    Si possono eseguire delle query molto più complesse attraverso il comando UNION, il quale permette di unire più query a patto che le colonne risultanti siano dello stesso tipo e abbiano lo stesso numero di colonne. Per verificare il numero di colonne si può eseguire il comando:

    • ‘ UNION SELECT NULL # se non da errore è presente una sola colonna.
    • ‘ UNION SELECT NULL,NULL;# se non da errore sono presenti due colonne.

    Una volta individuato il numero di colonne, se al posto di uno dei due NULL si inseriscono altri comandi, si possono estrarre informazioni di sistema, relative al database e altro:

    • @@version versione del database.
    • @@hostname hostname della macchina.
    • database() il nome del database.
    • schema_name from information_schema.schemata per tutti gli schemi del db.
    • ‘ union select null,table_name from information_schema.tables # per tutte le tabelle del db.
    • ‘ union select null,table_name from information_schema.tables where table_schema =’dvwa’ # per vedere le tabelle di un particolare schema
    • ‘ union select null,column_name from information_schema.columns where table_name =’users’ # colonne della tabella users.
    • ‘ union select user,password from users # per avere username e password contenuti nella tabella.

    -XSS

    Questa vulnerabilità permette di eseguire del codice javascript all’interno di applicazioni web. La configurazione del form è:

    echo '<pre>Hello ' . $_GET[ 'name' ] . '</pre>';

    Di conseguenza se inseriamo <script> alert("hackerato") </script> viene eseguito il codice javascript.
